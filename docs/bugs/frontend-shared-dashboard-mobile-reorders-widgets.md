# frontend: shared dashboard reorders widgets on mobile breakpoints

## Symptom

Opening a shared dashboard URL (`/shared/<token>`) on desktop showed widgets in
the designed order. On iPhone / narrow viewports (< 600px) the order changed:
all KPI cards got bucketed to the top regardless of design, and the remaining
widgets stacked in `widgets[]` array order rather than visual (`y`, `x`) order.

## Repro

1. Edit a dashboard so KPI cards sit *below* a chart (e.g. text/markdown intro
   on top, KPI strip in the middle, charts at the bottom).
2. Save and share.
3. Open the shared link on a viewport narrower than 600px.

Expected: same vertical order as the editor (intro → KPIs → charts).
Actual: KPIs jumped to row 0; everything else fell into array-order below.

## Root cause

`SocialDashboardGrid` ships separate layouts per breakpoint. The `lg` (≥600px)
layout uses the saved `(x, y)` directly. The `md` / `sm` / `xs` layouts were
generated by `buildCompactLayout`, which:

1. Split widgets into `cards = number-cards` and `charts = everything else`.
2. Placed all cards at `y=0`.
3. Then stacked charts in **array index order**, not designed `(y, x)` order.

So any widget that was a `number-card` was unconditionally promoted to the top,
and non-card widgets were laid out by their position in `widgets[]` — which is
usually the creation order, not the visual order the designer set.

## Fix

[frontend/src/features/studio/dashboard/buildCompactLayout.ts](frontend/src/features/studio/dashboard/buildCompactLayout.ts)
now sorts widgets by `(y, x)` and walks them once:

- Consecutive number-cards sharing a designed `y` form a single side-by-side row.
- Every other widget gets its own full-width row.
- Output order matches designed visual order.

Pure fn extracted out of `SocialDashboardGrid.tsx` so it can be unit-tested
without dragging in shadcn/Tailwind transitively.

## Regression test

[frontend/src/features/studio/dashboard/SocialDashboardGrid.test.ts](frontend/src/features/studio/dashboard/SocialDashboardGrid.test.ts)
— three cases: KPI between charts preserves order; consecutive KPIs stay
side-by-side; non-KPI widgets stack full-width in designed order.
